 실무에서도 비용 때문에 GPU를 무작정 늘리지 못하고 CPU 단계에서 어떻게든 병목을 해결해야 하는 상황이 차고 넘치거든요.
GPU가 없어도 참석자들이 **"와, Ray Data가 이래서 강력하구나!"**라고 감탄하게 만들 수 있는 CPU-Only 가성비 만점의 워크샵 시나리오 2가지를 짜드릴게요.
💡 GPU 없이 대박 터뜨리는 워크샵 시나리오 2선

### 시나리오 ①: "메모리(OOM) 폭발 지옥 탈출기" (강력 추천) ###
•	컨셉: "단돈 몇백 원짜리 CPU 노드에서 수십 GB의 대용량 이미지 처리하기"
•	실습 진행:
	1.	참석자들에게 일반 Python 코드(for 루프 + Pillow)로 S3에서 이미지 5,000장을 한 번에 읽어와 변형하는 코드를 돌리게 합니다. ➡️ 결과: EC2 메모리가 꽉 차서 프로세스가 터지거나(OOM), 맥없이 뻗어버립니다.
	2.	동일한 작업을 Ray Data의 Streaming 처리로 전환하게 합니다.
	3.	ray.data.read_images().map_batches()를 쓰는 순간, 메모리 그래프가 안정적인 일직선을 그리며 수천 장의 이미지가 끊김 없이 처리되는 기적을 보여줍니다.
•	메시지: "Ray Data는 GPU 전용 툴이 아닙니다. 제한된 CPU 자원(메모리) 속에서 거대한 데이터를 안전하게 흘려보내는(Streaming) 데이터 엔지니어링 툴입니다."

### 시나리오 ②: "순수 CPU 멀티코어 100% 갈아 넣기" (스케일아웃 체감) ###
•	컨셉: "내 EC2의 놀고 있는 8개 코어에게 진정한 분산 노동 시키기"
•	실습 진행:
	1.	참석자들에게 제공된 가성비 CPU EC2(예: 코어가 4개~8개 정도 있는 c6i.xlarge 등)를 활용합니다.
	2.	일반 파이썬 코드는 아무리 돌려도 코어 1개만 바쁘고 나머지는 노는 모습을 보여줍니다. (GIL의 한계)
	3.	Ray Data를 적용하는 순간, 내 인스턴스 안의 모든 CPU 코어가 동시에 100% 풀가동되며 S3에서 데이터를 병렬로 쫙쫙 긁어오는 장관을 htop이나 대시보드로 확인하게 합니다.
•	메시지: "EC2 인스턴스 하나의 잠재력을 극한으로 끌어올리는 법, 그리고 이 코드가 그대로 EKS로 가면 노드 100대짜리 스케일아웃으로 직결됩니다."

🛠️ 진행자를 위한 '완벽한 빌드업' 제안
참석자 실습에서 GPU를 못 주어 생기는 아쉬움은 강사님의 3부(EKS) 마무리에 딱 '한 대'의 GPU 데모를 얹는 것으로 완벽하게 해소할 수 있습니다.
•	참석자 (1부~2부): 비용이 아주 저렴한 CPU 전용 EC2에서 위 시나리오로 '메모리 절약'과 '멀티코어 병렬 처리'의 손맛을 깊게 배웁니다. (안 터지고 안전하게 도는 코딩 습관 습득)


### •	강사 데모 (3부): ###
"여러분들이 짠 그 CPU 전처리 코드 뒤에, 제가 EKS에 GPU 노드 딱 1대를 붙여서 AI 모델 추론 파이프라인을 연결해 보겠습니다" 하고 KubeRay 아키텍처의 완성형을 보여주는 것입니다.
이렇게 구성하면 AWS 비용은 CPU 가격만 나가므로 극도로 아끼면서도, 참석자들은 실무에서 당장 써먹을 수 있는 '가장 현실적인 데이터 엔지니어링 가이드'를 배워가게 됩니다.
가성비와 교육적 효과를 모두 잡은 완벽한 아키텍처 아닌가요? 걱정 말고 CPU로 밀어붙이셔도 됩니다!
